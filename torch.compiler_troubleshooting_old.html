作者：迈克尔·拉佐斯

注意

本文档已过时，现在主要作为如何运行 torch.compile minifier 的初级资源。请参阅更新的故障排除文档。同时，torch.compile 也提供了更全面的指南。

运行时错误诊断

从高层次来看，TorchDynamo 堆栈由 Python 代码的图捕获（TorchDynamo）和后端编译器组成。例如，后端编译器可能包括后向图跟踪（AOTAutograd）和图降级（TorchInductor）*。错误可能发生在堆栈的任何组件中，并将提供完整的堆栈跟踪。

确定错误发生的位置，您可以使用信息级别日志 torch._logging.set_logs(dynamo = logging.INFO) 或 TORCH_LOGS="dynamo" 并查找 Step #: ... 输出。日志在每个步骤的开始和结束时生成，因此错误应对应的步骤是最新的已记录步骤，其结束尚未被记录。步骤对应于堆栈的以下部分：

步骤

组件

1

TorchDynamo

2

编译器后端

3

TorchInductor

如果信息日志不足，您可以使用可用的后端选项。这些选项包括：

  • "eager" : 仅运行 TorchDynamo 的前向图捕获，然后使用 PyTorch 运行捕获的图。这可以提供 TorchDynamo 是否引发错误的指示。

  • "aot_eager" : 运行 TorchDynamo 捕获前向图，然后使用 AOTAutograd 跟踪后向图，而不需要任何额外的后端编译器步骤。然后使用 PyTorch 的 eager 模式运行前向和后向图。这有助于缩小问题范围到 AOTAutograd。

缩小问题的通用步骤如下：

  1. 使用 "eager" 后端运行您的程序。如果错误不再出现，问题在于所使用的后端编译器（如果使用 TorchInductor，请继续进行第 2 步。如果不使用，请参阅本节）。如果使用 "eager" 后端时错误仍然存在，则是在运行 torchdynamo 时出现错误。

  2. 此步骤仅在 TorchInductor 作为后端编译器时才需要。使用 "aot_eager" 后端运行模型。如果此后端引发错误，则错误发生在 AOTAutograd 跟踪期间。如果使用此后端时错误不再出现，则错误在于 TorchInductor*。

以下各节将分析这些情况。

注意

火炬电感器后端由 AOTAutograd 追踪和火炬电感器编译器本身组成。我们将通过引用 TorchInductor 作为后端，并将火炬电感器降级作为降低 AOTAutograd 追踪的图的阶段来消除歧义。

Torchdynamo 错误

如果生成的错误发生在 "eager" 后端，那么 TorchDynamo 很可能是错误的来源。以下是一个将生成错误的示例代码。

import torch

import torch._dynamo as dynamo


def test_assertion_error():
    y = torch.ones(200, 200)
    z = {y: 5}
    return z

compiled_test_assertion_error = torch.compile(test_assertion_error, backend="eager")

compiled_test_assertion_error()

如消息所示，您可以通过设置 torch._dynamo.config.verbose=True 来获取 TorchDynamo 错误和用户代码的完整堆栈跟踪。除了此标志外，您还可以通过 torch._logging.set_logs(dynamo = logging.INFO) 或 TORCH_LOGS="dynamo" 设置 TorchDynamo 的 log_level 。这些级别包括：

  • logging.DEBUG 或 TORCH_LOGS="+dynamo" ：打印遇到的每条指令以及以下所有日志级别。

  • logging.INFO ：打印编译的每个函数（原始和修改后的字节码）以及捕获的图，以及以下所有日志级别。

  • logging.WARNING （默认）：打印图断点以及以下所有日志级别。

  • logging.ERROR ：仅打印错误。

如果模型非常大，日志可能会变得难以处理。如果错误发生在模型 Python 代码的深处，仅执行发生错误的帧可以方便调试。有两个工具可以实现这一点：

  • 将环境变量 TORCHDYNAMO_DEBUG_FUNCTION 设置为所需的函数名，将只运行具有该名称的 torchdynamo 函数。

  • 启用记录/回放工具（设置 torch._dynamo.config.replay_record_enabled = True ），当遇到错误时将导出执行记录。然后可以回放此记录，仅运行发生错误的帧。

诊断 TorchInductor 错误

如果错误在没有使用 "eager" 后端时不会发生，那么后端编译器就是错误的来源（例如错误示例）。TorchDynamo 的后端编译器有多种选择，其中 TorchInductor 适合大多数用户的需求。本节以 TorchInductor 为例进行说明，但一些工具也可以与其他后端编译器一起使用。

下面是我们关注的堆栈部分：

使用 TorchInductor 作为后端，AOTAutograd 用于从 torchdynamo 捕获的前向图中生成反向图。需要注意的是，在跟踪过程中以及 TorchInductor 将前向和反向图降低到 GPU 代码或 C++时可能会发生错误。一个模型可能由数百或数千个 FX 节点组成，因此很难缩小到导致此问题发生的确切节点。幸运的是，有工具可以自动将这些输入图缩小到导致问题的节点。第一步是确定错误是在 AOTAutograd 跟踪反向图期间发生还是在 TorchInductor 降低期间发生。如上所述，在步骤 2 中，可以使用 "aot_eager" 后端单独运行 AOTAutograd 而不进行降低。如果使用此后端错误仍然发生，这表明错误发生在 AOTAutograd 跟踪期间。

下面是一个例子：

import torch

import torch._dynamo as dynamo

model = torch.nn.Sequential(*[torch.nn.Linear(200, 200) for _ in range(5)])

def test_backend_error():

    y = torch.ones(200, 200)
    x = torch.ones(200, 200)
    z = x + y
    a = torch.ops.aten._foobar(z)  # dummy function which errors
    return model(a)


compiled_test_backend_error = torch.compile(test_backend_error, backend="inductor")
compiled_test_backend_error()

如果你将 torch.compile(backend="inductor") 更改为 torch.compile(backend="aot_eager") ，它将无错误运行，因为问题出在 TorchInductor 降级过程中，而不是 AOTAutograd 中。

精简 TorchInductor 错误

从这里，让我们运行精简器以获取最小化复现。设置环境变量 TORCHDYNAMO_REPRO_AFTER="aot" （或直接设置 torch._dynamo.config.repro_after="aot" ）将生成一个 Python 程序，该程序将 AOTAutograd 生成的图减少到最小的子图，该子图可以复现错误。（下面是一个示例，其中我们精简了 TorchDynamo 生成的图）使用此环境变量运行程序应显示几乎相同的输出，并额外显示 minifier_launcher.py 已写入的位置。输出目录可以通过设置 torch._dynamo.config.base_dir 为有效的目录名称进行配置。最后一步是运行精简器并检查它是否成功运行。成功的运行看起来像这样。如果精简器成功运行，它将生成可运行的 Python 代码，该代码可以复现确切的错误。对于我们的示例，这是以下代码：

import torch
from torch import tensor, device
import torch.fx as fx
from torch._dynamo.testing import rand_strided
from math import inf
from torch.fx.experimental.proxy_tensor import make_fx

# torch version: 1.13.0a0+gitfddfc44
# torch cuda version: 11.6
# torch git version: fddfc4488afb207971c54ad4bf58130fdc8a4dc5


# CUDA Info:
# nvcc: NVIDIA (R) Cuda compiler driver
# Copyright (c) 2005-2022 NVIDIA Corporation
# Built on Thu_Feb_10_18:23:41_PST_2022
# Cuda compilation tools, release 11.6, V11.6.112
# Build cuda_11.6.r11.6/compiler.30978841_0

# GPU Hardware Info:
# NVIDIA A100-SXM4-40GB : 8

from torch.nn import *

class Repro(torch.nn.Module):
    def __init__(self):
        super().__init__()

    def forward(self, add):
        _foobar = torch.ops.aten._foobar.default(add);  add = None
        return (_foobar,)

args = [((200, 200), (200, 1), torch.float32, 'cpu')]
args = [rand_strided(shape, stride, dtype, device) for shape, stride, dtype, device in args]
mod = make_fx(Repro())(*args)
from torch._inductor.compile_fx import compile_fx_inner

compiled = compile_fx_inner(mod, args)
compiled(*args)

Repro 模块的 forward 方法包含导致问题的确切操作。在提交问题时要包括任何精简的复现，以帮助调试。

压缩后端编译器错误 ¶

对于除 TorchInductor 之外的后端编译器，查找导致错误的子图的过程几乎与 TorchInductor 中的错误处理过程相同，但有一个重要的注意事项。即，现在将在 TorchDynamo 跟踪的图上运行压缩器，而不是 AOTAutograd 的输出图。让我们通过一个例子来讲解。

import torch

import torch._dynamo as dynamo

model = torch.nn.Sequential(*[torch.nn.Linear(200, 200) for _ in range(5)])
# toy compiler which fails if graph contains relu
def toy_compiler(gm: torch.fx.GraphModule, _):
    for node in gm.graph.nodes:
        if node.target == torch.relu:
            assert False

    return gm


def test_backend_error():
    y = torch.ones(200, 200)
    x = torch.ones(200, 200)
    z = x + y
    a = torch.relu(z)
    return model(a)


compiled_test_backend_error = torch.compile(test_backend_error, backend=toy_compiler)
compiled_test_backend_error()

为了在 TorchDynamo 跟踪前向图后运行代码，您可以使用 TORCHDYNAMO_REPRO_AFTER 环境变量。使用 TORCHDYNAMO_REPRO_AFTER="dynamo" （或 torch._dynamo.config.repro_after="dynamo" ）运行此程序应产生以下输出，并在 {torch._dynamo.config.base_dir}/repro.py 中显示以下代码。

注意

另一个选项是 TORCHDYNAMO_REPRO_AFTER 为 "aot" ，这将运行压缩器在生成反向图之后。

import torch
import torch._dynamo as dynamo
from torch import tensor, device
import torch.fx as fx
from torch._dynamo.testing import rand_strided
from math import inf
from torch._dynamo.debug_utils import run_fwd_maybe_bwd

from torch.nn import *

class Repro(torch.nn.Module):
    def __init__(self):
        super().__init__()

    def forward(self, add):
        relu = torch.relu(add);  add = None
        return (relu,)


mod = Repro().cuda()
opt_mod = torch.compile(mod, backend="None")


args = [((200, 200), (200, 1), torch.float32, 'cpu', False)]
args = [rand_strided(sh, st, dt, dev).requires_grad_(rg) for (sh, st, dt, dev, rg) in args]


with torch.cuda.amp.autocast(enabled=False):
    ref = run_fwd_maybe_bwd(mod, args)
    res = run_fwd_maybe_bwd(opt_mod, args)

压缩器成功将图压缩到引发错误的操作 toy_compiler 。与 TorchInductor 错误处理流程的不同之处在于，压缩器在遇到后端编译器错误后自动运行。运行成功后，压缩器将 repro.py 写入 torch._dynamo.config.base_dir 。

性能分析 ¶

访问 TorchDynamo 分析器 ¶

TorchDynamo 内置了用于收集和显示每个编译阶段花费时间的统计函数。这些统计信息可以通过调用 torch._dynamo.utils.compile_times() 在执行 Torch._Dynamo 后访问。默认情况下，它返回一个字符串表示，显示了每个 TorchDynamo 函数按名称花费的编译时间。

使用 TORCH_COMPILE_DEBUG 进行 TorchInductor 调试

TorchInductor 内置了统计和跟踪功能，用于显示每个编译阶段花费的时间、输出代码、输出图可视化以及 IR 转储。这是一个旨在使理解并调试 TorchInductor 内部结构更加容易的调试工具。

让我们用一个以下测试程序（ repro.py ）来运行一个示例：

import torch

@torch.compile()
def test_model(x):
    model = torch.nn.Sequential(
        torch.nn.Linear(10, 10),
        torch.nn.LayerNorm(10),
        torch.nn.ReLU(),
    )
    return model(x)


y = test_model(torch.ones(10, 10))

设置环境变量 TORCH_COMPILE_DEBUG=1 将导致创建一个调试跟踪目录，默认情况下，此目录将位于当前目录，并命名为 torch_compile_debug（这可以在 torchdynamo 配置字段 debug_dir_root 和 env var TORCH_COMPILE_DEBUG_DIR 中覆盖）。在此目录内，每次运行将有一个单独的文件夹，该文件夹以运行的时间戳和进程 ID 命名：

在运行文件夹中，将有一个 torchdynamo 目录，其中包含调试日志，以及一个 torchinductor 文件夹，其中包含每个编译内核的子文件夹，用于存储电感调试工件。

进一步进入 torchinductor 目录， \*.log 文件是编译 AOT Autograd 阶段的日志， model__0_forward_1.0 包含电感调试工件。

下面是内容摘要：

  • fx_graph_readable.py 和 fx_graph_runnable.py 是电感接收到的 fx_graph 的可读和可运行版本。

  • fx_graph_transformed.py 是电感器运行所有效果后 fx 图。

  • ir\*.txt 是电感器融合前后的 ir。

  • output_code.py 是子图编译后的 triton 内核。

图断点

给定如下程序：

def some_fun(x):
    ...

compiled_fun = torch.compile(some_fun, ...)
...

TorchDynamo 将尝试将 some_fun 中的所有 torch/tensor 操作编译成一个单一的 FX 图，但它可能无法将所有内容都捕获到一个图中。

一些图断裂的原因对 TorchDynamo 来说是无法克服的，并且难以修复。 - 调用 torch 之外的 C 扩展对 torchdynamo 来说是不可见的，并且可能执行任意操作，而 TorchDynamo 无法引入必要的保护措施（参见《让 Dynamo 声音响亮：保护措施》）以确保编译后的程序可以安全重用。如果生成的片段很小，图断裂可能会影响性能。为了最大化性能，尽可能减少图断裂是很重要的。

识别图断裂的原因

要识别程序中的所有图断点及其原因，可以使用 torch._dynamo.explain 。该工具在提供的函数上运行 TorchDynamo 并汇总遇到的图断点。以下是一个示例用法：

import torch
import torch._dynamo as dynamo
def toy_example(a, b):
    x = a / (torch.abs(a) + 1)
    print("woo")
    if b.sum() < 0:
        b = b * -1
    return x * b
explanation = dynamo.explain(toy_example)(torch.randn(10), torch.randn(10))
print(explanation_verbose)
"""

输出包括：

  • out_guards - 一个列表的列表，其中每个子列表包含必须通过的守卫，以确保跟踪的图有效。

  • graphs - 成功跟踪的图模块列表。

  • ops_per_graph - 一个列表的列表，其中每个子列表包含在图中运行的运算。

要在遇到第一个图断点时抛出错误，请使用 fullgraph 模式。此模式禁用 TorchDynamo 的 Python 回退，并且只有当整个程序可转换为单个图时才成功。示例用法：

def toy_example(a, b):
   ...

compiled_toy = torch.compile(toy_example, fullgraph=True, backend=<compiler>)(a, b)

过度重新编译

当 TorchDynamo 编译一个函数（或其一部分）时，它会针对局部变量和全局变量做出某些假设，以便允许编译器优化，并将这些假设表达为在运行时检查特定值的守卫。如果这些守卫中的任何一个失败，Dynamo 将重新编译该函数（或其一部分）多达 torch._dynamo.config.recompile_limit 次。如果您的程序正在达到缓存限制，您首先需要确定哪个守卫失败以及您的程序中的哪个部分触发了它。

如果您的程序具有有限度的动态性，您可能能够调整 TorchDynamo 的缓存限制，以便为每个变体编译和缓存，但如果缓存限制过高，您可能会发现重新编译的成本超过了任何优化好处。

torch._dynamo.config.recompile_limit = <your desired cache limit>

TorchDynamo 计划支持许多常见的动态张量形状的情况，例如变化的批量大小或序列长度。它不计划支持秩动态性。在此期间，可以通过与分桶技术协调使用，设置特定的缓存限制，以实现某些动态模型可接受的重新编译次数。

准确性调试 ¶

如果您设置环境变量 TORCHDYNAMO_REPRO_LEVEL=4 ，可以减少精度问题，它的工作方式与 git bisect 模型类似，完整的重现可能如下 TORCHDYNAMO_REPRO_AFTER="aot" TORCHDYNAMO_REPRO_LEVEL=4 我们需要这样做的原因是下游编译器会生成代码，无论是 Triton 代码还是 C++后端，这些下游编译器的数值可能在细微之处有所不同，但会对您的训练稳定性产生重大影响。因此，精度调试器对我们检测代码生成或后端编译器中的错误非常有用。

如果您想确保 torch 和 triton 之间的随机数生成相同，则可以启用 torch._inductor.config.fallback_random = True

扩展调试 ¶

可以通过以下实验性标志启用扩展调试。

TORCHDYNAMO_EXTENDED_DEBUG_GUARD_ADDED - 如果守卫的字符串表示与该标志值匹配，则提供扩展调试信息。例如，将其设置为“Ne(s0, 10)”以在守卫发出时生成完整的 Python 和 C++回溯。 TORCHDYNAMO_EXTENDED_DEBUG_CREATE_SYMBOL - 当特定符号分配时提供扩展调试信息。例如，将其设置为“u2”以在创建此符号时生成完整的 Python 和 C++回溯。 TORCHDYNAMO_EXTENDED_DEBUG_CPP - 为所有扩展调试设置以及错误提供扩展调试信息（C++回溯）。例如，将其设置为“1”。C++回溯速度慢且非常频繁，因此默认情况下不包括在扩展调试中。

冷启动时间及缓存损坏调试

为了测量冷启动编译时间或调试缓存损坏，可以通过传递 TORCHINDUCTOR_FORCE_DISABLE_CACHES=1 或设置 torch._inductor.config.force_disable_caches = True 来覆盖任何其他缓存配置选项并禁用所有编译时间缓存。
