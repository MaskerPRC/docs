全局变量桥接:
    处理收集 Callgrind 统计信息时（某些）全局变量的转移。

关键要点：传递的全局变量必须包装在`CopyIfCallgrind`中。
使用 `Timer.collect_callgrind`。

考虑以下代码片段：
    ```
导入 pickle 模块
导入 timeit 模块

        class Counter:
            value = 0

            def __call__(self):
                self.value += 1

        counter = Counter()
        timeit.Timer("counter()", globals={"counter": counter}).timeit(10)
        print(counter.value)  # 10

        timeit.Timer(
"counter()"
            globals={"counter": pickle.loads(pickle.dumps(counter))}
        ).timeit(20)
print(counter.value)  # 仍然是 10
    ```

在第一种情况下，`stmt` 使用 `globals` 中的对象执行
然而，添加序列化和反序列化改变了
语义和可能有意义地改变行为。

这是一个在收集 Callgrind 统计信息时的实际考虑。
与基于 `exec` 的执行（`timeit` 在底层使用）不同
可以与调用者共享内存数据结构，Callgrind 收集
需要完全新的过程才能在 Valgrind 下运行。这意味着
任何用于语句执行的数据结构都必须
序列化和反序列化在子进程中执行。

为了避免（用户不可见）过程中出现的令人惊讶的语义
边界，可以通过 `globals` 传递的内容受到严格限制
对于 `Timer.collect_callgrind`。预期大多数设置应该是可实现的（尽管可能不太方便），通过传递一个 `setup` 字符串。
可实现的（尽管可能不太方便）。
。

然而，也有一些例外。其中一类是 TorchScripted 函数。
因为它们需要一个具体的源代码文件，所以无法使用`setup`字符串来定义。
另一类是 torch.nn.Modules，它们的构建可能很复杂，难以强制执行。
其构造可能很复杂，难以强制执行，且成本高昂。
将 `setup` 字符串。最后，大多数内置类型已经足够好
表现良好且足够常见，因此也应允许。
`globals={"n": 1}`非常方便。)

幸运的是，所有这些都有明确定义的序列化语义。此类
负责启用 Valgrind 子进程使用元素
只要它们是允许的类型。

用户需通过封装来确认此序列化
